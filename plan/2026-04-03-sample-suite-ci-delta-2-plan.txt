## Task Summary
- 本轮只修 `sample-suite-eval` 第二轮 CI 新暴露的 4 个真实阻塞点，不扩新范围：
  1. Windows 仍未采纳 Visual Studio bundled LLVM，且无条件 choco 路径污染了 `LLVM_DIR`。
  2. macOS workflow 漂到 Homebrew `llvm 22.x`，pass plugin 头文件契约失效。
  3. Linux/Android 脚本仍把 host-built helper/runtime 当成可跨 target 复用的事实，导致 Android 用户态链路没有真正用目标工具链闭环。
  4. iOS 样例仍链接 host-built helper/runtime，存在 target/SDK/ABI 错配风险。
- 本轮目标是恢复 `sample-suite-eval` 全绿，并把 sample-suite 的 host/plugin 构建、target helper 构建、样例最终链接边界彻底冻结，避免第三次返工。

- Windows 继续 fail-closed：没有 `clang-cl` 或没有有效 LLVM CMake dir 就直接失败。

### Failure Modes And Error Handling
- Windows
  - workflow preflight 先验证 bundled LLVM；若缺失，再受控安装并验证 choco fallback。
  - bundled 与 fallback 都无效时，直接 fail。
  - 若 `clang-cl` 不存在，直接 fail。
- macOS
  - 若 `brew --prefix llvm@18` 失败，直接 fail。
  - 若 `LLVM_DIR` 不含 `LLVMConfig.cmake` 或 `PassPlugin.h`，直接 fail。
- Linux/Android
  - 若 NDK target compiler 不存在，直接 fail。
  - 若任一 helper object 未成功生成，直接 fail。
  - 若 `android_so` 仍落回 host clang，直接 fail。
  - 若 `android_ko` 未使用 Android target clang driver + `-nostdlib -fuse-ld=lld -Wl,-r` 完成 relocatable link，直接 fail。
- iOS
  - 若 iOS helper object 未按 iOS target + SDK 生成，直接 fail。

### Regression Perimeter
- 直接改动面
  - workflow 依赖安装与环境导出
  - Windows/macOS/Linux-Android 三个平台样例构建脚本
- 邻接面
  - `eippf_cc.py` 现有参数契约
  - `string_token_runtime.cpp` 作为 helper 源的编译兼容性
  - sample-suite 输入目录结构
- 必须保持不变
  - sample ids 不变
  - `protect-and-evaluate` job 不变
  - evidence layout 不变
- 第二序风险
  - iOS helper 之前被 host-built archive 掩盖，修完 plugin 后可能暴露 ABI/SDK 链接问题
  - Android user-mode 改用 NDK compiler 后，kernel `.ko` 若误用 host `ld` 会直接再次失败
  - Windows workflow 去掉无条件 choco 后，若脚本没有真正采纳 VS LLVM，会立即暴露

### Unchanged Guarantees
- sample-suite 仍覆盖全部类型。
- 保护 pass 仍然对 C/C++ 样例生效。
- `artifact_shape/signed_policy` 评估契约不变。
- `protect-and-evaluate` 仍在 Linux evaluator 上构建保护工具并消费 3 平台 artifact。

### Probable Failure Scenarios
- 只修 workflow 不修脚本，仍会在平台脚本内部复现旧错误。
- Android user-mode 若继续用 host clang，仅靠 `--target` 仍可能再次落回 host runtime/linker 行为。
- `android_ko` 若把 aarch64 object 继续交给 host `ld -r`，会直接引入第三轮 CI 失败。
- Android user helper 若缺 `-fPIC`，会在 shared object 链接期失败。
- Android kernel helper 若缺 `-ffreestanding/-fno-builtin/-fno-stack-protector/-fno-asynchronous-unwind-tables`，会在 `-nostdlib` 链接期暴露未解析符号。
- iOS 若继续链接 host-built helper，plugin 修好后会在最终链接阶段因 SDK/ABI 不匹配失败。

### Task-Specific Coding Rules
- 不改 `eippf_cc.py`。
- 不新增通用 runtime archive 协议。
- helper 只能在各平台脚本里显式编译，并放在各自 sample output tree 的私有中间目录。
- Windows 规则
  - workflow 不再无条件安装 choco LLVM；必须先做 bundled LLVM preflight。
  - `build_windows.ps1` 的 LLVM 解析顺序固定为：
    1. `env:EIPPF_VS_LLVM_DIR` 有效则使用，并写 `llvm_source=vs_bundled`
    2. `env:LLVM_DIR` 有效且其 canonical path 等于 `C:\\Program Files\\LLVM\\lib\\cmake\\llvm` 时才允许使用，并写 `llvm_source=choco_fallback`
    3. 其余来源一律拒绝并失败；禁止 `llvm-config --cmakedir`、禁止 `derived-from-clang-cl`、禁止采纳任意外部 `env:LLVM_DIR`
  - helper 直接按本计划冻结的两条命令编译为 `string_token_runtime.windows.user.obj` 与 `string_token_runtime.windows.sys.obj`，并参与 exe/dll/sys 链接；不再依赖 CMake 产出的 `eippf_string_token_runtime.lib`
- Linux/Android 规则
  - Linux host plugin 固定使用 workflow 导出的 `EIPPF_HOST_LLVM_ROOT=/usr/lib/llvm-18` 与 `LLVM_DIR=/usr/lib/llvm-18/lib/cmake/llvm`，禁止脚本自行通过 `llvm-config` 或 PATH 发现 host LLVM。
  - Linux `linux_elf/linux_so` helper 固定使用 host clang++ + `-fPIC` 编译。
  - Linux `.ko` helper 固定使用 host freestanding clang++ 编译，必须满足 kernel-safe flags。
  - Android `android_so` 固定使用 workflow 导出的 `EIPPF_ANDROID_NDK_ROOT` 下的 NDK `aarch64-linux-android24-clang/clang++` 构建；wrapper 调用前必须切换 `EIPPF_CLANG/EIPPF_CLANGXX`。
  - Android `android_so` helper 固定使用 `-fPIC`。
  - Android `.ko` helper 与模块 object 固定使用 workflow 导出的 `EIPPF_ANDROID_NDK_ROOT` 下的 Android target clang/clang++ + Android sysroot 编译；最终链接固定使用 Android target clang driver + `-nostdlib -fuse-ld=lld -Wl,-r`。
- macOS/iOS 规则
  - workflow 只能安装 `llvm@18`；脚本不得自己 fallback 到 `llvm 22.x`。
  - pass plugin 仍由 `llvm@18` host clang++ 构建。
  - iOS helper 固定使用 `xcrun --sdk iphonesimulator --find clang++` 与最终 iOS sample 完全相同的 `-target` 与 `-isysroot` 编译。
  - iOS helper 固定使用 `-fPIC`。

- Windows 路径解析只允许 Visual Studio 安装树与 workflow preflight 导出的受控 fallback；禁止其他外部 `LLVM_DIR`。

## Decision Log
- D1
  - 旧假设“host-built helper/runtime 可以跨 Linux/Android/iOS 复用”被 run `23950090370` 否定。
  - 新决策：sample helper 改为 target-specific object，由各平台脚本按最终 target toolchain 直接编译。
- D2
  - Windows 不再无条件安装 choco LLVM。
  - 新决策：默认使用 Visual Studio bundled LLVM；workflow 先做 preflight，仅 bundled 缺失时才启用受控 choco fallback；脚本负责解析并验证最终 CMake dir。
- D3
  - macOS 不再使用 `brew install llvm`。
  - 新决策：统一固定 `llvm@18`，并把版本固定责任放在 workflow。
- D4
  - Android `.ko` 不再调用 host `ld -r`。
  - 新决策：固定使用 Android target clang driver + `-nostdlib -fuse-ld=lld -Wl,-r` 完成 relocatable link。
- D5
  - iOS helper 不再允许“Homebrew LLVM clang++ 或 xcrun 二选一”。
  - 新决策：helper 必须固定用 `xcrun --sdk iphonesimulator --find clang++`。
- D6
  - Android helper 不再共用一份 object。
  - 新决策：`android_so` 和 `android_ko` 分别使用 user helper object 与 kernel helper object；前者强制 `-fPIC`，后者强制 kernel-safe flags。
- D7
  - 平台报告不再共用同名文件。
  - 新决策：三份 `toolchain_report` 改为固定路径、固定键名、固定 workflow 校验命令，避免下载合并时覆盖与人工验收漂移。
- D8
  - 用户已显式授权本轮修改 `/.github/workflows/sample-suite-eval.yml` 与使用真实多 agent 并行。
  - 新决策：本轮把 workflow 视为治理/验证资产；仓库 traceability 仍按 `Planner / Engineer / Security Reviewer` 三角色登记，工程实现分 4 个 Engineer 子轨，审核分 3 个 reviewer checkpoint。
