Architecture Contract:
- `eippf_vm_rt` 是独立可消费的 VM runtime 静态库。
- 凡是被编入 `eippf_vm_rt` 的对象文件，其直接链接期实现闭包必须由 `eippf_vm_rt` 自己闭合。
- 本次明确修 `eippf_vm_rt` 的 link surface：
  - 修补位置：`/workspace/core/runtime/CMakeLists.txt`
  - 修补方式：在 `eippf_vm_rt` 自身目标定义里补齐 `backend_policy` 实现闭包。
  - 本次固定方案：把 `src/backend_policy.cpp` 纳入 `eippf_vm_rt`，不要求编码 agent 自行比较其他方案。
- 本次明确不修 `tests/CMakeLists.txt`：
  - 原因 1：`/workspace/core/tests/CMakeLists.txt` 中 `vm_runtime_bridge_vm2_test -> eippf_vm_rt` 的链接关系本身是正确的消费者合同。
  - 原因 2：若改测试去额外链接 `eippf_runtime`，只会让测试通过，但生产库 `eippf_vm_rt` 仍然不是自闭合 runtime 交付物。
  - 原因 3：把缺口补在测试侧会掩盖真实发布面回归，违反 fail-closed。
- 本次明确不采用 `eippf_vm_rt -> eippf_runtime` 的宽表面修法：
  - 原因 1：这会把 VM runtime 依赖面扩到整个 `eippf_runtime`，不属于最小 delta。
  - 原因 2：`eippf_runtime` 已自带 `secure_ir_engine.cpp`，宽链接会放大耦合与未来重复接线风险。
  - 原因 3：当前 blocker 只缺 `backend_policy.cpp`，不应引入更大依赖图。
- 本次保持以下不变：
  - `MemoryHAL::runtime_dynamic_code_allowed()` 的语义不变。
  - `backend_policy.cpp` 的 target dispatch / JIT / executable-pages gate 语义不变。
  - `secure_ir_engine.cpp` 的运行时行为不变。

Decision Log:
- `DL-P3C-01`
  - 决定：根因定位为 `eippf_vm_rt` 的生产目标未闭合 backend policy helper 的实现。
  - 原因：`secure_ir_engine.cpp` 通过 `MemoryHAL::runtime_dynamic_code_allowed()` 依赖 `target_kind_supports_desktop_jit()` 与 `target_forbids_runtime_executable_pages()`，而这两个符号只在 `backend_policy.cpp` 中定义。
- `DL-P3C-02`
  - 决定：本 delta 必须修 `eippf_vm_rt` 的 link surface。
  - 原因：blocker 发生在生产库 `eippf_vm_rt` 被消费者链接时，不在测试入口本身。
- `DL-P3C-03`
  - 决定：本 delta 不修 `/workspace/core/tests/CMakeLists.txt`。
  - 原因：现有测试已经正确暴露生产库 contract 缺口；改测试会把生产问题伪装成测试问题。
- `DL-P3C-04`
  - 决定：本 delta 不采用 `eippf_vm_rt -> eippf_runtime` 的宽表面连接。
  - 原因：这不是最小闭口，会扩大依赖面，并把本应由 `eippf_vm_rt` 自身承担的实现闭包外包给上层库。
- `DL-P3C-05`
  - 决定：安全验收必须显式覆盖 JIT gate 与 safe-target。
  - 原因：本 blocker 正好穿过 `MemoryHAL` 与 backend policy helper，若只看“链接能过”不足以证明安全合同未回退。
