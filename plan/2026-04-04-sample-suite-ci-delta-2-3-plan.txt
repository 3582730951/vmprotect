## System Boundaries
- Windows:
  - workflow 负责显式 provision 和验证 LLVM18 root
  - `build_windows.ps1` 只消费已验证的 `LLVM_DIR` / `EIPPF_ALLOWED_LLVM_ROOT`
- Android:
  - wrapper 只用于 host LLVM18 的 compile-only 保护编译
  - NDK `clang++` 只负责 object-only 最终链接
- iOS:
  - wrapper 只用于 host LLVM18 的 compile-only 保护编译
  - `xcrun clang` 只负责 object-only 最终链接
- validator 必须同时证明：
  - 保护编译阶段确实发生在 LLVM18 兼容编译器上
  - 最终链接阶段不再包含 source 输入，也不再触发 pass plugin 注入

## Architecture Contract
- Windows 不再做 candidate-root 猜测式发现；改成“显式 provision pinned LLVM18 + 固定 root 验证”。
- Android/iOS 都采用两段式：
  1. `protected compile`
     - 使用 wrapper
     - compile-only
     - 含 source 输入
     - 由 host LLVM18 compiler 执行
     - 允许注入 pass plugin
     - 产出单独的 protected object
  2. `final link`
     - 使用 target-native driver 直接执行，不走 wrapper
     - object-only
     - 输入为 `protected object + helper object`
     - 不再含 source 输入
     - 不得出现 `-fpass-plugin` / wrapper path / plugin path
- 这套方案利用 `eippf_cc.py` 现有事实：
  - compile-only 或 source 输入才注入 pass plugin
  - object-only link 不会注入 pass plugin
- 所有 report/validator 必须升级到能表达“compile provenance”和“link provenance”分离。
- Linux/Android validator 中现有 `check_single_input()` 只保留给 `linux_elf/linux_so/linux_ko`。
- Android `android_so/android_ko` 必须改成新的双输入校验：
  - `link_inputs=<protected_o>;<helper_o>`
  - final link 不再跑 `enforce_link_language_token_state_machine()`
  - 新增 compile-only 专用状态机函数用于 `*_protected_compile_command`
- macOS/iOS validator 中现有 `ios_link_inputs` 单输入 helper-only 契约必须删除。
- iOS final link 不再跑 `enforce_link_language_token_state_machine()`；改为新增 compile-only 专用状态机函数校验 `ios_protected_compile_command`。
- compile-only 专用状态机函数固定接受：
  - `command_text`
  - `command_key`
  - `source_path`
  - `output_path`
- compile-only 专用状态机函数固定规则：
  - 允许 `-o <protected>.o` 或 `-o<protected>.o`
  - 不把 `-o` 的值视为“非法 object/archive token”
  - 不需要 `helper_path`
  - 必须有 `-c`
  - 必须有 `-x c`
  - source token 必须跟在 `-x c` 之后

## Invariants
- `.ko` 现有 `-no-pie` 规则必须保留，不允许回退。
- Windows validator 仍必须验证：
  - `LLVMConfig.cmake`
  - compiler provenance
  - plugin build sidecar
  - helper compile sidecar
- Android/iOS helper 编译路径不变：
  - Android helper 继续走 target-native NDK clang++
  - iOS helper 继续走 `xcrun clang++`
- wrapper 不修改；通过脚本重排 compile/link 阶段来规避 ABI 崩溃。
- Android/iOS final link validator 固定为“双输入 object-only 契约”，不得保留旧的 helper-only 单输入断言。
- Android/iOS protected compile validator 固定承担 compile-only 专用状态机校验，而不是复用现有 link 状态机函数。

## Failure Modes And Error Handling
- Windows:
  - pinned LLVM18 缺任一关键文件，workflow 立即 fail
  - 不允许 fallback 到 VS bundled LLVM 或 preinstalled LLVM20
- Android/iOS:
  - protected compile 输出缺失，脚本立即 fail
  - final link 若仍含 source token、wrapper path、plugin path，validator fail
  - link inputs 数量或顺序错误，validator fail

## Probable Failure Scenarios
- Windows pinned install 仍然得到错误版本或 stale root，导致 validator 通过但 build 失败
- Android/iOS final link 仍然通过 wrapper 或仍带 source 输入，导致 plugin 重新注入
- protected compile 输出路径未进入 report/validator，导致 validator 自身失真
- link inputs 顺序不对，helper 或 protected object 丢失
- `.ko` no-pie 新规则被 Android split-link 改坏

## Task-Specific Coding Rules
- Windows preflight 禁止继续使用 candidate-root 猜测逻辑。
- Windows 必须显式 provision pinned LLVM18，并验证：
  - `bin\\clang-cl.exe`
  - `lib\\cmake\\llvm\\LLVMConfig.cmake`
  - `include\\llvm\\Passes\\PassPlugin.h`
- Android/iOS protected compile 必须：
  - 使用 wrapper
  - 使用 host LLVM18 compiler
  - compile-only
  - 带 `-x c`
  - 带 source path
  - 产出独立 protected object
- Android/iOS final link 必须：
  - 直接调用 target-native driver
  - object-only
  - 禁止 source path
  - 禁止 wrapper path
  - 禁止 plugin path
  - 禁止 `-fpass-plugin`
- validator 必须用显式路径、显式 token、显式 input count/order 做校验，不能只做 substring。
