Architecture Contract
- 双视图是硬契约：
  - `artifact_shape` 只回答“保护工具是否把产物做坏/是否完成字符串压制/是否满足形状约束”。
  - `signed_policy` 只回答“需要签名的四类目标在 CI fixture + trusted verifier 下是否满足严格策略”。
- `implementation_defect` 与 hard-fail 必须分离：
  - `implementation_defect` 进入 `summary.json` 与 `report.html`。
  - hard-fail 只用于：
    - 输入缺失
    - 命令执行失败
    - manifest/summary 契约破坏
    - required output 缺失
    - trusted verifier/fixture signer 基础设施损坏
- reverse evidence layout 必须同时满足两类消费者：
  - 新布局：`original.file.txt / original.strings.txt / original.readelf.txt / original.objdump.txt`
  - 兼容布局：`reverse/<sample-id>/strings.txt` 与 `reverse/<sample-id>/metadata.txt`
  - `metadata.txt` 至少包含：
    - `sample_id`
    - `input_relpath`
    - `protected_relpath`
    - `input_sha256`
    - `output_sha256`
    - `validation_scope`
    - `known_limits`
- kernel mutator 语义固定为：
  - `.ko` 成功路径必须基于合法 `ET_REL` + section table fixture。
  - `.ko` 验收看 `.note.eippf` 注入，不再看 EOF trailer。
  - `.sys` 仍看 overlay trailer，不改 section table。
- sample build integration 固定为：
  - Linux/Android/iOS/Windows 样例都先构建 `eippf_protection_suite_pass`。
  - wrapper 必须拿到平台对应的 pass plugin 产物。
  - Windows 缺 `clang-cl` 直接失败，不回退 `cl.exe`。
- full-chain 验收固定分层：
  - 本地：脚本/测试/summary/report 契约闭环。
  - CI：跨平台样例编译 + protect/evaluate + evidence upload 闭环。
