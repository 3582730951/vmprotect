# 2026-04-02 Phase 5 `.sh launcher` 详细计划

## Execution Mode
- `real-multi-agent`

## Assurance Level
- `heavy`

## Task Charter
- Objective:
  - 把当前仓库中的 `core/script_guard/` 从“只会生成加密 bundle 的 skeleton”推进到 `bundle -> launcher -> external key provider -> no-persistent-plaintext -> artifact audit -> perf` 的完整闭环。
- Scope:
  - `/workspace/core/script_guard/`
  - `/workspace/core/tools/artifact_audit.py`
  - `/workspace/core/tests/script_guard/`
  - `/workspace/core/tests/tools/artifact_audit_strict_test.cpp`
  - `/workspace/core/tests/perf/`
  - `/workspace/core/tests/CMakeLists.txt`
  - `/workspace/plan/`
- Non-Goals:
  - 不扩展到 `Phase 6 Android .dex`。
  - 不扩展到 `Phase 7 iOS`。
  - 不把 `.sh` 保护伪装成与 native VM/JIT 同等级动态对抗。
  - 不支持需要 `source` 外部脚本、依赖 `$0/BASH_SOURCE` 自省、主动 `set -x` 的脚本进入 release lane。
- Constraints:
  - PM 负责设计、切分、验收，工程 agent 不做架构决策。
  - `.sh` lane 必须 `fail-closed`。
  - 不允许持久化明文脚本到磁盘。
  - 不允许嵌入 key material。
  - 只能依赖当前仓库已有 contract：`shell_ephemeral` / `shell_launcher`。
- Acceptance Criteria:
  - 生成 `shell bundle v3`。
  - 新增独立 `launcher` 可消费 bundle + manifest + external key provider。
  - 启动链对缺 key、坏 key、坏 bundle、坏 manifest、unsafe shell 特性全部 `fail-closed`。
  - `artifact_audit --strict` 对 shell lane 有独立 gate。
  - 至少具备：
    - 1 条 success test
    - 1 条 failure test
    - 1 条 security-edge test
    - 1 条 perf smoke test
  - `configure/build/ctest/perf_smoke` 通过。
  - 启动额外开销 `<=250ms`，重复执行整体开销 `<=10%`。

## Architecture Contract
### 1. Lane 定义
- `target_kind` 固定为 `shell_ephemeral`
- `backend_kind` 固定为 `shell_launcher`
- `runtime_lane` 固定为 `shell_launcher`
- `mutation_profile` 固定为 `shell_bundle`
- `signature_policy` 固定为 `required_verifier`
- `allow_jit=false`
- `allow_runtime_executable_pages=false`
- `allow_persistent_plaintext=false`
- `require_fail_closed=true`
- `plaintext_ttl_ms=0`

### 2. `.sh` 威胁模型
- 静态 grep / 字符串定位
  - 威胁：
    - `strings`
    - `grep`
    - 教程锚点定位
    - 关键命令/内部标识搜索
  - 防护：
    - bundle 中不允许出现明文脚本 anchor
    - manifest 不记录明文脚本摘要、原始 key、原始脚本内容
    - `artifact_audit` 对 denylist string 保持严格扫描
- `set -x`
  - 威胁：
    - 执行期把命令明文打到 stderr/xtrace
  - 防护：
    - bundler 在加密前做 plaintext lexical scan
    - 发现 `set -x` 或 `set -o xtrace` 直接拒绝打包
    - launcher 运行前清理 `PS4`、`BASH_XTRACEFD`、`SHELLOPTS`
- `source`
  - 威胁：
    - 外部明文脚本依赖破坏 no-persistent-plaintext
    - 外部 include 链难以审计
  - 防护：
    - bundler 发现 `source ` 或行首 `. ` 直接拒绝打包
    - release lane 只支持 self-contained 单文件 shell script
- 子 shell
  - 威胁：
    - 子 shell 继承环境或消费解密后的脚本输入
  - 防护：
    - launcher 不导出 key 到环境变量
    - launcher 使用匿名 pipe 把明文送入解释器 stdin，不生成明文文件路径
    - 父进程写完后立即擦除 plaintext buffer 并关闭 pipe
- 环境变量泄漏
  - 威胁：
    - `BASH_ENV`
    - `ENV`
    - `PROMPT_COMMAND`
    - `PS4`
    - `LD_PRELOAD`
    - `DYLD_*`
  - 防护：
    - launcher 在 child exec 前执行固定 allowlist 环境重建
    - 严格清理上述变量
- 临时文件恢复
  - 威胁：
    - `/tmp` 残留
    - shell 临时展开文件
    - 意外明文 sidecar
  - 防护：
    - Phase 5 release lane 只允许 `pipe_stdin_exec`
    - 不创建 `.plain/.tmp/.sh.dec` 文件
    - tests 必须扫描 temp dir，确认无明文副产物
    - `artifact_audit` 必须把 `plaintext_output=false` 作为 shell gate 条件
- provider 退化为静态 key file
  - 威胁：
    - `--key-provider=<path>` 直接指向磁盘上的明文 key file
    - key material 落到仓库工作目录或脚本 temp dir
  - 防护：
    - `--key-provider=<path>` 只允许代表受控 provider 端点：
      - `executable_adapter`
      - `fifo`
      - `unix_socket`
    - 普通文件 provider、指向普通文件的 symlink provider 一律 `fail-closed`
    - provider 必须返回协议化 response，原始 key material 不得持久化到仓库工作目录或脚本 temp dir
    - provider 端点消费后不得在 bundle、manifest、stdout、stderr、env、temp file 中留下原始 key material

### 3. 交付物形态
- 产物 1：
  - `eippf_script_guard`
  - 职责：读取明文 `.sh`，扫描 unsafe 特性，读取 external key provider，输出 `bundle v3 + manifest`
- 产物 2：
  - `eippf_script_launcher`
  - 职责：读取 `bundle + manifest + external key provider`，校验 contract，匿名解密，pipe 注入解释器执行
- 产物 3：
  - `artifact_audit --strict`
  - 职责：验证 shell bundle invariants + manifest contract + no-persistent-plaintext 证据

### 4. `shell bundle v3` 固定格式
- magic:
  - `ESHB`
- format_version:
  - `3`
- key_material_marker:
  - 固定 `0`
- flags:
  - bit0 `shebang_present`
- interpreter_tag:
  - 允许值仅：
    - `sh`
    - `bash`
    - `dash`
- key_id_len
- payload_len
- key_id bytes
- encrypted payload bytes

### 5. Manifest 固定字段
`shell bundle` manifest 必须输出以下字段，字段名不允许改：
- `schema_version: 2`
- `kind: "shell_script_bundle"`
- `target_kind: "shell_ephemeral"`
- `backend_kind: "shell_launcher"`
- `runtime_lane: "shell_launcher"`
- `mutation_profile: "shell_bundle"`
- `signature_policy: "required_verifier"`
- `artifact_kind: "shell_bundle"`
- `allow_jit: false`
- `allow_runtime_executable_pages: false`
- `allow_persistent_plaintext: false`
- `require_fail_closed: true`
- `plaintext_ttl_ms: 0`
- `loader_format_version: 3`
- `key_provider_protocol: "eippf.external_key.v1"`
- `key_provider_endpoint_kind: "executable_adapter" | "fifo" | "unix_socket"`
- `key_provider_static_file: false`
- `key_id: <non-empty>`
- `key_material_embedded: false`
- `execution_model: "pipe_stdin_exec"`
- `launcher_host: "linux_posix"`
- `interpreter_tag: "sh" | "bash" | "dash"`
- `trace_env_scrubbed: true`
- `source_policy: "self_contained_only"`
- `unsafe_shell_features: []`
- `plaintext_output: false`
- `no_persistent_plaintext_goal: true`

### 6. Launcher 消费链
- 输入：
  - `--input-bundle=<path>`
  - `--manifest=<path>`
  - `--key-provider=<path>`
  - `--key-id=<id>`
  - `--` 后面为传给脚本的参数
- 顺序：
  1. 读取 manifest
  2. 读取 bundle header
  3. 校验 manifest 与 bundle header 的：
     - `target_kind`
     - `backend_kind`
     - `runtime_lane`
     - `mutation_profile`
     - `key_id`
     - `key_material_embedded=false`
     - `execution_model=pipe_stdin_exec`
  4. 读取 external key provider
  5. 解密到内存 buffer
  6. `fork + pipe + execv`
  7. child:
     - 清理环境
     - `dup2(pipe_read, STDIN_FILENO)`
     - `execv(interpreter, [interpreter, "-s", "--", user_args...])`
  8. parent:
     - 把 plaintext 写入 pipe
     - 立即擦除 plaintext buffer
     - 等待 child
     - 返回 child exit code
- fail-closed 条件：
  - manifest 缺字段
  - bundle header 非 v3
  - key provider 不可读/格式坏/拒绝/key_id 不匹配
  - interpreter_tag 不在 allowlist
  - 检测到 unsafe shell 特性
  - 任何 pipe/fork/exec 失败

### 7. External Key Provider Contract
- `--key-provider=<path>` 的 `<path>` 只允许代表以下 3 类 provider 端点之一：
  - `executable_adapter`
    - `<path>` 必须是可执行 adapter 路径
    - 只能直接 `execve/execv` 调用，禁止 shell 拼接、禁止 `/bin/sh -c`
    - adapter 必须通过 `stdout` 返回 `eippf.external_key.v1` 协议化 response
  - `fifo`
    - `<path>` 必须是受控 named pipe
    - 只允许一次性读取消费协议化 response
  - `unix_socket`
    - `<path>` 必须是 Unix domain socket
    - 通过固定 request/response 协议读取协议化 response
- 明确禁止：
  - 普通文件形式的静态 key file
  - 指向普通文件的 symlink provider
  - 仓库工作目录中的明文 key 文件
  - 脚本 temp dir 中的明文 key 文件
  - bundle/manifest 嵌入原始 key material

## Open Risks
- `pipe_stdin_exec` 与脚本直传路径执行不同，依赖 `$0/BASH_SOURCE` 的脚本必须拒绝，否则会产生语义偏差。
- BusyBox `sh`、`dash`、`bash` 的细微差异可能导致脚本兼容性不一致；Phase 5 先按 Linux 常见 POSIX shell 收口。
- lexical scan 是保守策略，可能拒绝一部分本可运行的脚本，但这符合“保护优先、fail-closed”。
- `required_verifier` 依赖现有外部 verifier 流程；若 verifier 缺失，严格审计必须失败，不能降级。
