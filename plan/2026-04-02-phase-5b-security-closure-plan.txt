Task Charter:
- Objective:
  - 基于 Phase 5a 被 `Security Reviewer` BLOCK 的结论，以最小 delta 关闭 Phase 5 `.sh launcher` 的剩余安全闭环，明确补齐 `launcher manifest full-contract validation` 与 `runtime unsafe-shell recheck/fail-closed`，并同步冻结 `stderr leak closure` 与 `shell strict audit true execution`。
- Scope:
  - 把 Phase 5 计划要求的 shell manifest 固定契约字段全部纳入 launcher runtime 校验面。
  - 在 launcher 解密后、exec 前增加 runtime 级 unsafe-shell 复核，正面挡住 tampered bundle/manifest 绕过。
  - 关闭 executable provider `stderr` 泄漏面，保持 fail-closed。
  - 把 shell strict audit 从“只写 fixture”改成真实执行并做精确 token 断言。
- Non-Goals:
  - 不新增 manifest schema 字段。
  - 不新增 provider endpoint kind。
  - 不改 `bundle v3` 格式。
  - 不扩到 Phase 6。
  - 不重做 Phase 5 架构，不回写执行记录，不引入新运行模型。
- Acceptance Criteria:
  - launcher 对 Phase 5 shell manifest 固定字段全集做 exact-value runtime 校验，而不是只校验当前已消费字段。
  - launcher 在 decrypt 后、exec 前对明文脚本做 runtime unsafe-shell recheck；tampered bundle/manifest 无法绕过。
  - executable provider 只允许 `stdout` 协议输出；任意 `stderr` 字节都 fail-closed 且不向父进程可见输出泄漏。
  - `artifact_audit_strict_test` 必须真实执行 shell strict audit，并精确命中 success/failure token。
- Constraints:
  - 只能做 Phase 5b 最小修补。
  - 不改变 `shell_ephemeral -> shell_launcher -> pipe_stdin_exec` 边界。
  - 不放宽 fail-closed、no-persistent-plaintext、provider allowlist。
- Forbidden Shortcuts:
  - 不允许只补测试、不补 runtime 校验。
  - 不允许只信任 bundler 侧 unsafe-shell 扫描，不做 launcher runtime 复核。
  - 不允许通过吞掉 `stderr` 但继续成功来伪造泄漏闭环。
  - 不允许继续只写 shell audit fixture 而不实际执行 `run_audit(... --strict)`。

Regression Perimeter:
- Root Cause:
  - Phase 5a 只把 launcher 验证补到“已消费字段”，没有把 Phase 5 manifest 固定契约整体纳入 runtime 校验。
  - bundler-only 的 unsafe-shell 扫描无法覆盖 bundle/manifest 被篡改后的运行时绕过。
  - executable provider child 继承 `stderr` 仍有泄漏面。
  - shell strict audit 测试写了 shell fixture，但没有真实执行 shell strict audit。
- Changed Path:
  - launcher manifest parser/validator
  - launcher decrypt 后到 exec 前的安全复核
  - executable provider child I/O handling
  - shell launcher / shell audit tests
- Adjacent Paths:
  - bundle header version/schema cross-check
  - provider endpoint allowlist
  - bundler happy path
  - artifact audit shell token attribution
- Unchanged Guarantees:
  - `bundle v3` 不变
  - `pipe_stdin_exec` 不变
  - provider allowlist 不扩
  - no-persistent-plaintext 不放宽
  - fail-closed 不放宽
- Second-Order Risks:
  - full-contract 如果只 parse 不 assert，tampered manifest 仍可穿过。
  - runtime recheck 如果放在 exec 之后或依赖 manifest 放行，会失去 tamper 防线。
  - `stderr` 封堵如果只吞日志不 fail，会把协议违规伪装成成功。
  - shell strict audit 如果仍是 fixture-only，token 断言不可采信。
- Regression Matrix:
  - `script_guard_smoke_test`：bundler 基线保持
  - `script_launcher_success_test`：合法 happy path 保持
  - `script_launcher_failure_test`：manifest full-contract 单字段变异全部失败
  - `script_launcher_security_edge_test`：tampered bundle/manifest 的 unsafe-shell 绕过全部 fail-closed
  - `artifact_audit_strict_test`：shell strict success/failure token 来自真实执行
  - `shell_launcher_perf_smoke_test` / `perf_smoke`：性能预算继续 <10%

Unchanged Guarantees:
- launcher 仍只通过匿名 pipe 将明文脚本送入解释器 stdin。
- launcher 仍不通过 env/argv/temp file 传播 key material。
- bundler 的 pre-bundle lexical redlight 继续保留；Phase 5b 只是补 runtime 复核，不替代 bundler gate。

Second-Order Risks:
- tamper 测试如果直接把 bundle 改坏，可能只验证到“格式损坏”而不是“unsafe-shell 被 runtime 拦截”。
- shell strict fixture 如果同时命中多个 strict token，容易造成目标 token 归因不清。
- `stderr` 捕获如果 pipe/wait 顺序错误，可能引入死锁或截断。

2. Task Card 2: Runtime Unsafe-Shell Recheck / Fail-Closed
- Owner: `Agent 2 (Engineer)`
- Files:
  - `/workspace/core/script_guard/src/launcher.cpp`
  - `/workspace/core/tests/script_guard/script_launcher_security_edge_test.cpp`
- Change:
  - 在 decrypt 成功之后、`fork + execv` 之前，launcher 必须复用共享 `scan_unsafe_shell_features()` 对明文脚本做 `runtime unsafe-shell recheck/fail-closed`。
  - launcher 不得信任 manifest 的 `unsafe_shell_features` 作为 greenlight；manifest 只提供附加 redlight。
  - runtime 扫描结果非空时，launcher 必须在解释器启动前直接失败，不得进入 child exec。
- Required Cases:
  - 不能依赖 bundler 生成 unsafe bundle，因为 bundler 本来就该拒绝；必须从 safe bundle 出发做 tamper probes。
  - 在 `script_launcher_security_edge_test.cpp` 中至少新增以下四类 payload tamper case，并保持 manifest 仍声称 `unsafe_shell_features: []`：
    - 解密后脚本含 `set -o xtrace`
    - 解密后脚本含行首 `. `
    - 解密后脚本含 `$0`
    - 解密后脚本含 `${BASH_SOURCE`
  - 另增 tampered manifest case：
    - `unsafe_shell_features` 被改为非空数组
    - `unsafe_shell_features` 被改为非法类型
- Done When:
  - tampered bundle/manifest 的 unsafe-shell 绕过在 launcher runtime 被稳定 fail-closed。
  - 失败路径在 exec 前终止，不产生脚本输出，不进入解释器。

3. Task Card 3: Executable Provider `stderr` Leak Closure
- Owner: `Agent 2 (Engineer)`
- Files:
  - `/workspace/core/script_guard/src/external_key_provider.cpp`
  - `/workspace/core/tests/script_guard/script_launcher_success_test.cpp`
  - `/workspace/core/tests/script_guard/script_guard_smoke_test.cpp`
- Change:
  - `read_provider_response_from_executable()` 增加独立 `stderr` 捕获，不再让 provider child 继承父进程 `stderr`。
  - provider 协议只允许 `stdout` 返回；`stderr` 任意非空都视为协议违规并 fail-closed。
  - 捕获到的 `stderr` 内容不得透传到父进程 `stdout`、`stderr`、bundle、manifest、输出文件、临时副产物。
- Required Cases:
  - 保留一个无 `stderr` 的 executable provider success case，证明 happy path 不退化。
  - 新增一个 `stderr` secret marker case：
    - provider 在 `stdout` 返回合法 key 协议
    - provider 在 `stderr` 输出 secret marker
    - bundler/launcher 必须失败
    - marker 不得出现在任何父可见输出
- Done When:
  - `stderr leak closure` 以“捕获 + 不转发 + fail-closed”落地。
  - 不允许通过“静默吞掉 `stderr` 但继续成功”冒充闭环。
