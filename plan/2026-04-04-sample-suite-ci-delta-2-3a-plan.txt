## Task Summary
- 本轮只修 `/workspace/.github/workflows/sample-suite-eval.yml` 里的 3 个 validator/contract gap：
  1. Linux/Android 通用路径存在性循环错误地把 `android_so_link_inputs`、`android_ko_link_inputs` 当单一路径校验
  2. macOS/iOS 通用路径存在性循环错误地把 `ios_link_inputs` 当单一路径校验
  3. macOS/iOS 的 `enforce_compile_only_language_token_state_machine()` 强度低于 Android 同类实现，没有强制 `source token` 必须紧跟 `-x c`

## Objective

- iOS compile-only 状态机必须与 Android 同类实现保持同强度
- 不允许通过删除通用路径循环或扩大白名单“洗绿”

## Architecture Contract
- Linux/Android 通用路径存在性循环继续保留；仅把 `android_so_link_inputs`、`android_ko_link_inputs` 纳入“非单路径字段”排除集合
- macOS/iOS 通用路径存在性循环继续保留；仅把 `ios_link_inputs` 纳入“非单路径字段”排除集合
- iOS compile-only 状态机必须加严到 Android 同强度：
  - `-x c` 后的下一个 token 必须是 source token
  - 命令结束时若仍在等待 source token，直接失败
- 不改 Android compile-only 状态机
- 不改任何错误口径以外的无关逻辑

## Quality Gates
- `python3 - <<'PY' ... yaml.safe_load(...) ... PY` 能成功解析 `.github/workflows/sample-suite-eval.yml`
- `git diff --check` 无输出
- 静态复核满足：
  - Linux/Android 通用路径循环排除集合包含 `android_so_link_inputs`、`android_ko_link_inputs`
  - macOS/iOS 通用路径循环排除集合包含 `ios_link_inputs`
  - iOS `enforce_compile_only_language_token_state_machine()` 中存在 `pending_source_after_x_c` 或等价立即相邻状态变量
  - iOS 状态机同时覆盖：
    - 下一 token 不是 source token 的失败分支
    - 命令结束时仍缺失紧邻 source token 的失败分支
