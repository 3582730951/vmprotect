# 2026-04-02 Phase 4 / driver kernel-safe 详细计划

依附关系:
- 本文件依附于 `/workspace/plan/2026-04-02-phase-3-final-freeze-execution.txt`。
- 本文件依附于 `/workspace/plan/2026-04-02-terminal-goal-superseding-plan.txt`。
- 本文件只覆盖 `Phase 4` 的三条 driver/kernel-safe lane：
  - Windows `.sys`
  - Linux `.ko`
  - Android `.ko`
- 本文件不回头重做 `Phase 3`。
- 本文件不扩展到 `iOS` / `.dex` / `.sh`。

Execution Mode:
- `real-multi-agent`
- 这里的 `real-multi-agent` 指 `Phase 4` 实施时必须按仓库既定 3 角色独立执行，不把本轮文档起草过程伪装成“已经发生真实子代理调用”。
- 固定角色模型只允许：
  - `Agent 1 (Planner / PM)`
  - `Agent 2 (Engineer)`
  - `Agent 3 (Security Reviewer)`
- `Agent 1` 只写计划、冻结范围、下发 task cards、执行验收，不写实现代码。
- `Agent 2` 只按 task cards 修改自己被分配的文件，不自定义范围、不改计划。
- `Agent 3` 独立审核安全、gate、perf、fail-closed，不写实现代码。

Assurance Level:
- `heavy`

Task Charter:
- Objective:
  - 在 `Phase 3` 已冻结前提下，把 `Windows .sys`、`Linux .ko`、`Android .ko` 三条 lane 推进到 `Phase 4` 可执行、可验证、可审计状态。
  - 本批不是“补方向”或“补 skeleton”，而是直接闭合三条 kernel lane 的端到端证据链。
  - 必须同时闭合：
    - manifest 元数据
    - `artifact_audit --strict` gate
    - no-JIT / no-RWX / no-runtime-codegen 证据
    - sign-after-mutate 元数据与真实性严格失败链
    - 3 lane x 3 case 测试矩阵
    - perf 预算证据
- Scope:
  - 只允许覆盖以下交付面：
    - `core/post_link_mutator/*` 中与 kernel lane manifest 直接相关的部分
    - `core/tools/artifact_audit.py`
    - `core/tests/*` 中与 kernel lane runtime evidence、lane closure、perf evidence 直接相关的部分
  - 本批 scope 明确只限：
    - Windows `.sys`
    - Linux `.ko`
    - Android `.ko`
  - 本批不允许把 scope 外溢到：
    - `iOS`
    - `Android .dex`
    - `.sh`
    - `Phase 3` user-mode lane
- Non-Goals:
  - 不扩展到 `iOS App Store` lane。
  - 不扩展到 `Android .dex` lane。
  - 不扩展到 `.sh` lane。
  - 不回头重做 `Phase 3` 已冻结 user-mode 主线。
  - 不把 `Phase 4` 降级成“contracts + audit + tests skeleton”。
  - 不在本批重构 wrapper、pass、`ip_weaver_ir`、string mainline。
- Acceptance Criteria:
  - 配置与构建：
    - `cmake -S /workspace/core -B /workspace/build_phase4 -G Ninja`
    - `cmake --build /workspace/build_phase4 -j`
  - runtime hard-gate：
    - `ctest --test-dir /workspace/build_phase4 --output-on-failure -R 'backend_policy_test|runtime_backend_extraction_contract_test|runtime_memory_hal_windows_driver_safe_test|runtime_memory_hal_linux_kernel_module_safe_test|runtime_memory_hal_android_kernel_module_safe_test|runtime_secure_ir_engine_windows_driver_safe_test|runtime_secure_ir_engine_linux_kernel_module_safe_test|runtime_secure_ir_engine_android_kernel_module_safe_test|hardened_target_windows_driver_runtime_kind_test|hardened_target_linux_kernel_module_runtime_kind_test|hardened_target_android_kernel_module_runtime_kind_test|hardened_target_windows_driver_secure_ir_engine_test|hardened_target_linux_kernel_module_secure_ir_engine_test|hardened_target_android_kernel_module_secure_ir_engine_test|jit_enclave_pipeline_gate_test'`
  - mutator / manifest / audit：
    - `ctest --test-dir /workspace/build_phase4 --output-on-failure -R 'post_link_mutator_smoke_test|artifact_audit_strict_test'`
  - lane tests：
    - `ctest --test-dir /workspace/build_phase4 --output-on-failure -R 'windows_driver_success_test|windows_driver_failure_test|windows_driver_security_edge_test|linux_kernel_module_success_test|linux_kernel_module_failure_test|linux_kernel_module_security_edge_test|android_kernel_module_success_test|android_kernel_module_failure_test|android_kernel_module_security_edge_test'`
  - perf：
    - `ctest --test-dir /workspace/build_phase4 --output-on-failure -R 'windows_driver_perf_smoke_test|linux_kernel_module_perf_smoke_test|android_kernel_module_perf_smoke_test'`
    - `cmake --build /workspace/build_phase4 --target perf_smoke`
  - `artifact_audit --strict` 验收结果必须满足：
    - three success lanes: `strict_failures=[]`
    - `windows_driver` success: `validation_mode=external_verifier`
    - `linux_kernel_module` success: `validation_mode=external_verifier`
    - `android_kernel_module` success: `validation_mode=external_verifier`
    - `windows_driver` failure: 必须命中 `signature_authenticity_missing`、`signature_verifier_failed`、`signature_verifier_untrusted`、`signature_verifier_digest_mismatch`、`signature_authenticity_rejected` 中至少一项
    - `linux_kernel_module` failure: 必须命中 `vermagic_mismatch` 或 `signature_missing`
    - `android_kernel_module` failure: 必须命中 `gki_kmi_mismatch` 或 `signature_missing`
    - three security-edge lanes: 必须命中 `kernel_gate_failed`
  - success manifest / audit report 必须包含：
    - `runtime_lane=kernel_safe`
    - `mutation_profile=kernel_module`
    - `signature_policy=sign_after_mutate`
    - `sign_after_mutate_required=true`
    - `kernel_compat_profile`
    - `allow_jit=false`
    - `allow_runtime_executable_pages=false`
    - `allow_persistent_plaintext=false`
    - `require_fail_closed=true`
    - lane-specific compat flag：
      - Windows `.sys` -> `hvci_profile=true`
      - Linux `.ko` -> `vermagic_profile=true`
      - Android `.ko` -> `gki_kmi_profile=true`
  - perf 预算必须满足：
    - hot path `<=8%`
    - overall `<=12%`
- Constraints:
  - driver lane 硬约束必须写死且可被 tests / audit 观察到：
    - compile-time `allow_jit=false`
    - compile-time `allow_rwx=false`
    - `no runtime codegen`
    - `sign-after-mutate required`
    - Windows `.sys` 需要 `hvci_profile`
    - Linux `.ko` 需要 `vermagic_profile`
    - Android `.ko` 需要 `gki_kmi_profile`
  - 所有 kernel lane 必须 fail-closed。
  - authenticity/verifier 严格失败条件不得降级、不得静默忽略。
- Forbidden Shortcuts:
  - 禁止把已有 `kernel_safe_backend` descriptor 直接当成 `Phase 4` 完成证据。
  - 禁止只补 manifest 字段，不把字段接入 `artifact_audit --strict`。
  - 禁止只补 tests skeleton，不闭合 strict gate 和 metadata。
  - 禁止为通过 perf gate 放宽 no-JIT / no-RWX / no-runtime-codegen。
  - 禁止改写 `Phase 3` 已冻结范围。
  - 禁止在 dirty worktree 中顺手回滚或覆盖他人改动。
