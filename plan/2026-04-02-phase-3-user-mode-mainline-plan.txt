# 2026-04-02 Phase 3 user-mode mainline detailed plan

Execution Mode:
- `real-multi-agent`

Assurance Level:
- `heavy`

Task Summary:
- `Phase 3` 只覆盖三条 user-mode 发布级主线：
  - Windows `.exe/.dll`
  - Linux `ELF/.so`
  - Android `.so`
- 目标不是给现有产物继续追加 trailer，而是把当前仓库推进到“可发布的 user-mode 保护主线”：
  - 字符串保护与 lexical anchor 治理进入统一主线
  - `PE/ELF` post-link mutate 从 `trailer-only` 升级到 publish-grade mutate baseline
  - Android `.so` threat hardening baseline 进入真实计划与验收
  - native artifact audit、平台证据、性能门完整闭环

Task Charter:
- Objective:
  - 基于已冻结的 `Phase 2C`，完成 `Windows .exe/.dll`、`Linux ELF/.so`、`Android .so` 三条 user-mode 发布级主线的详细实施计划，并为后续工程轨提供无歧义 task cards。
- Scope:
  - `core/wrapper/`
  - `core/passes/`
  - `core/runtime/`
  - `core/post_link_mutator/`
  - `core/tools/artifact_audit.py`
  - `core/tests/`
  - `plan/`
- Non-Goals:
  - 不扩到 `.sys/.ko`
  - 不扩到 `.dex`
  - 不扩到 `.sh`
  - 不扩到 iOS
  - 不把 `JITEnclavePass` 直接当作 `Phase 3` 的 release-grade protection body
- Constraints:
  - 先计划、再三审、再编码。
  - PM 不接管实现代码。
  - 工程 agent 只按固定文件白名单与步骤执行。
  - `Windows/Linux/Android .so` 的最终性能预算 `<=10%`。
- Acceptance Criteria:
  - `build/test/perf_smoke` 通过。
  - 产出至少一条 `Windows .exe/.dll` 证据、一条 `Linux ELF/.so` 证据、一条 `Android .so` 证据。
  - `native artifact audit` 对样本产物给出可审计输出。
  - `strings / lexical anchor / internal tutorial strings` 有统一治理门。
  - Android `.so` threat hardening baseline 覆盖：
    - `JNI symbol`
    - `strings`
    - `anti-hook`
    - `anti-debug`
  - `post_link_mutator` 不再只是 `trailer-only`。
- Forbidden Shortcuts:
  - 不能把 `trailer-only` 伪装成 publish-grade mutate。
  - 不能把 `JITEnclavePass` 的 dummy payload 路径当成 release-grade body。
  - 不能为通过性能门而降低 critical string / lexical anchor 保护强度。
  - 不能删除 `smoke` 或把 audit 失败降格为 warning。
  - 不能把 Android `.so` 风险留在“以后再测”。

Architecture Contract Summary:
- 发布级 user-mode 主线固定为：
  - `wrapper -> passes -> runtime support -> post_link_mutator -> artifact_audit -> evidence`
- `JITEnclavePass` 在 `Phase 3` 中只保留为 desktop-only gated experiment：
  - 不能作为 release-grade 主 body
  - 不能成为验收主证据
  - 真实发布主线以 `StringProtection + CFF/MBA + IAT minimization + post-link mutate + audit` 为主
- 字符串保护是一等目标，统一合同如下：
  - 编译期抽取字符串候选
  - lexical anchor denylist / tutorial-anchor denylist 必须进入主门禁
  - 运行期只允许最短生命周期 decode
  - Android JNI / exported symbol / tutorial strings 进入同一治理面
- `PE/ELF` publish-grade mutate baseline 固定包含：
  - 产物分类后的 user-mode only 路径
  - import / dynamic symbol 审计与收敛
  - section / note / metadata mutate baseline
  - fail-closed mutation result
  - manifest / audit 一致性
- Android `.so` hardening baseline 固定包含：
  - JNI export surface 收敛
  - lexical anchor / strings residual gate
  - 反调试与 hook 风险基线
  - 至少一条 Android `.so` 证据样本

Decision Log:
- `DL-P3-01`
  - 决定：`Phase 3` 不把 `JITEnclavePass` 作为 release-grade 主路径
  - 原因：当前实现仍使用 dummy payload 路径，不能作为真实保护主体验收
- `DL-P3-02`
  - 决定：`StringProtectionPass` 与 lexical anchor audit 组成统一主线
  - 原因：用户已明确字符串定位是第一攻击面，且当前仓库已有 pass 与 denylist/audit 基础
- `DL-P3-03`
  - 决定：`PE/ELF` mutate 与 `artifact_audit` 必须并行设计，不能先变异后补审计
  - 原因：当前仓库已有 `post_link_mutator` 和 `artifact_audit`，但都还未形成发布级 user-mode 证据闭环
- `DL-P3-04`
  - 决定：Android `.so` 单独保留 threat hardening baseline 轨
  - 原因：`JNI symbol / strings / anti-hook / anti-debug` 不能靠 Windows/Linux 轨顺带覆盖
