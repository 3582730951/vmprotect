## Root Cause Judgment
- 根因固定为：
  1. 当前 pinned Windows LLVM18 组合下，`add_llvm_pass_plugin(...)` 的 `MODULE` 路径不可用。
  2. `core/passes/CMakeLists.txt` 仍然无条件对这些 pass target 执行 `target_include_directories` / `target_compile_definitions` / `eippf_apply_common_compile_options`。
  3. 结果是 configure 阶段在 pass plugin target-type 上 fail，而不是等到样例编译阶段才 fail。
- 次级风险固定为：
  1. 即便改成 Windows `SHARED` fallback，若不显式处理 DLL 导出与输出目录，仍可能在下一步卡在：
     - `eippf_protection_suite_pass.dll` 未导出 `llvmGetPassPluginInfo`
     - `eippf_protection_suite_pass.dll` 未落到 `sample_suite_inputs/pass_plugins`

## Architecture Contract
### A. 新增统一兼容层
- 新增 `core/cmake/PassPluginCompat.cmake`
- 只暴露一个 helper：
  - `eippf_add_pass_plugin_target(<name> SOURCES <src...> [COMPILE_DEFINITIONS <def...>] [OUTPUT_NAME <value>])`

### B. helper 的唯一行为
- helper 内部必须按以下顺序执行：
  1. 计算 `name_upper`
  2. 强制 `LLVM_${name_upper}_LINK_INTO_TOOLS=OFF`
  3. 判断是否进入 Windows fallback：
     - 条件固定为：`WIN32`
  4. 若不进入 fallback（即非 Windows）：
     - 调用 `add_llvm_pass_plugin(${name} ...)`
  5. 若进入 fallback（即 Windows）：
     - 先尝试 `TARGET LLVM`
     - 若 `TARGET LLVM` 不存在，则要求 `COMMAND llvm_map_components_to_libnames`
     - 若两者都不存在，直接 `message(FATAL_ERROR ...)`
     - 使用 `add_library(${name} SHARED ...)`
     - 链接顺序固定为：
       - 若 `TARGET LLVM` 存在：`target_link_libraries(${name} PRIVATE LLVM)`
       - 否则：`llvm_map_components_to_libnames(eippf_pass_plugin_llvm_libs Core Passes Support Analysis TransformUtils ScalarOpts InstCombine ipo Instrumentation IRReader)`，再 `target_link_libraries(${name} PRIVATE ${eippf_pass_plugin_llvm_libs})`
     - `set_target_properties(${name} PROPERTIES PREFIX "" WINDOWS_EXPORT_ALL_SYMBOLS ON)`
  6. 不论哪条路径，都必须立刻读取 `TARGET TYPE`
  7. 如果 target type 不是可编译目标，直接 `message(FATAL_ERROR ...)`
  8. 再统一追加：
     - include dirs
     - compile definitions
     - `eippf_apply_common_compile_options`
     - output properties

### C. helper 的 target-type gate
- helper 内部允许的 target type 只有：
  - `MODULE_LIBRARY`
  - `SHARED_LIBRARY`
- 以下类型一律视为 blocker：
  - `INTERFACE_LIBRARY`
  - `UTILITY`
  - `OBJECT_LIBRARY`
  - `STATIC_LIBRARY`
  - 未定义

### D. helper 的输出目录契约
- Windows fallback 下必须把 DLL 输出强制对齐到 sample-suite 现有契约。
- 规则固定为：
  - 在 `WIN32` 路径中无条件检查 `CMAKE_LIBRARY_OUTPUT_DIRECTORY*`
  - 若 `CMAKE_LIBRARY_OUTPUT_DIRECTORY` 非空，则同步设置：
    - `RUNTIME_OUTPUT_DIRECTORY "${CMAKE_LIBRARY_OUTPUT_DIRECTORY}"`
    - `LIBRARY_OUTPUT_DIRECTORY "${CMAKE_LIBRARY_OUTPUT_DIRECTORY}"`
  - 若 `CMAKE_LIBRARY_OUTPUT_DIRECTORY_RELEASE` 非空，则同步设置：
    - `RUNTIME_OUTPUT_DIRECTORY_RELEASE "${CMAKE_LIBRARY_OUTPUT_DIRECTORY_RELEASE}"`
    - `LIBRARY_OUTPUT_DIRECTORY_RELEASE "${CMAKE_LIBRARY_OUTPUT_DIRECTORY_RELEASE}"`
- 不修改 `build_windows.ps1` 的 configure 参数。

### E. suite plugin 专属契约
- `eippf_protection_suite_pass` 仍固定：
  - `OUTPUT_NAME "eippf_protection_suite_pass"`
  - `PREFIX ""`
- `if(NOT EIPPF_PASS_PLUGIN)` 的注册逻辑保留。
- 最终 Windows 仍必须让 `build_windows.ps1` 在：
  - `sample_suite_inputs/pass_plugins/eippf_protection_suite_pass.dll`
  上找到真实插件产物。
- 本 delta 的导出 gate 固定由现有 Windows 样例编译承担：
  - `build_windows.ps1` 在 plugin 构建完成后会立刻通过 wrapper 使用 `--pass-plugin <...dll>` 编译 `windows_exe/windows_dll/windows_sys`
  - 若 `llvmGetPassPluginInfo` 未被正确导出或插件无法被 Clang 加载，本 delta 必须在 `Build Windows Samples` 阶段 fail，而不是继续进入 validator
