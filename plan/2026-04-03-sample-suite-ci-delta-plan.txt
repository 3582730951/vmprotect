## Task Summary
- 本轮只修 sample-suite CI 新暴露的 4 个已坐实根因，不扩新范围：
  1. Windows workflow 预构建 pass plugin 使用失效的 `LLVM_DIR`。
  2. macOS workflow 预构建 pass plugin 没把 Homebrew LLVM clang 绑定到 CMake。
  3. Linux native sample 经过 pass 改写后，链接缺少 `eippf_string_token_*` runtime。
  4. workflow 与平台脚本重复构建 pass plugin，导致平台逻辑分叉。
- 本轮目标是恢复 `sample-suite-eval` 的跨平台闭环，并把平台样例构建链收敛成单一事实来源。

## Architecture Contract
- 单一事实来源决策
  - 选择：`sample build scripts` 成为各平台 `plugin + runtime + sample` 构建的单一事实来源。
  - 理由：
    - 当前 3 个 CI 根因里，Windows 与 macOS 都来自 workflow 预构建逻辑和脚本逻辑分叉。
    - Linux/macOS/Windows 的平台差异应只在各自脚本里维护，workflow 只负责安装前置依赖并调用脚本。
    - `protect-and-evaluate` job 例外：它在 Linux evaluator 上构建 `eippf_post_link_mutator` / `eippf_dex_toolchain` / `eippf_script_guard`，这是评估工具链，不属于平台 sample 产物构建，允许保留。
- 最小 runtime 注入决策
  - 新增或抽出最小静态库 target：`eippf_string_token_runtime`
  - 该 target 只包含 `src/string_token_runtime.cpp`，不链接 `dl`，不带 user-mode 后端，不带 attestation，不带 VM runtime。
  - 样例构建只注入这个最小库，不注入整包 `eippf_runtime`。
- native sample auto-link 决策
  - 当前不扩展 `eippf_cc.py` CLI；runtime 注入通过“脚本把最小 runtime 静态库路径作为正常 linker 输入显式附加到 wrapper 编译命令或最终链接命令”落地。
  - 所有平台脚本 configure plugin/runtime build dir 时统一传：
    - `-DCMAKE_ARCHIVE_OUTPUT_DIRECTORY=<out>/runtime_libs`
  - 最小 runtime 静态库输出路径固定为：
    - Linux/macOS: `<out>/runtime_libs/libeippf_string_token_runtime.a`
    - Windows: `<out>/runtime_libs/eippf_string_token_runtime.lib`
  - user-mode 样例：
    - `linux_elf`
    - `linux_so`
    - `android_so`
    - `windows_exe`
    - `windows_dll`
    - `ios_macho`
  - kernel/driver 样例：
    - `linux_ko`
    - `android_ko`
    - `windows_sys`
  - 对于 user-mode，脚本调用 wrapper 时在编译/链接命令尾部显式追加最小 runtime 静态库路径：
    - Linux/macOS/Android Clang 风格：
      - `python3 eippf_cc.py --pass-plugin <plugin> -- <src...> -o <out> <runtime-lib>`
    - Windows clang-cl 风格：
      - `python eippf_cc.py --pass-plugin <plugin> -- /nologo ... <src> /link /OUT:<out> <runtime-lib>`
  - 对于 kernel/driver，脚本的最终链接命令必须显式带上最小 runtime 静态库：
    - Linux/Android: `ld -r ... <runtime-lib>`
    - Windows: `link/lld-link ... <runtime-lib>`
- toolchain 绑定决策
  - Linux/macOS 脚本在 CMake configure pass/runtime build dir 时，必须显式传：
    - `-DCMAKE_C_COMPILER=<HOST_CLANG>`
    - `-DCMAKE_CXX_COMPILER=<HOST_CLANGXX>`
  - Windows 脚本在 CMake configure 时，必须显式传：
    - `-DCMAKE_C_COMPILER=<clang-cl>`
    - `-DCMAKE_CXX_COMPILER=<clang-cl>`
    - `-DLLVM_DIR=<validated-llvm-cmake-dir>`
- Windows LLVM 解析决策
  - `build_windows.ps1` 的 LLVM dir 解析顺序固定为：
    1. 若环境变量 `LLVM_DIR` 存在且目录内包含 `LLVMConfig.cmake`，使用它。
    2. 否则若 `llvm-config.exe` 可执行，使用 `llvm-config --cmakedir`，并验证 `LLVMConfig.cmake`。
    3. 否则从检测到的 `clang-cl` 根目录派生 `../lib/cmake/llvm`，并验证 `LLVMConfig.cmake`。
    4. 三者都失败则明确 fail，打印已尝试路径。
  - Windows 判废标准固定为：
    - 只允许 `clang-cl` 作为 C/CXX 编译器。
    - 只允许上述受控 `LLVM_DIR` 解析顺序。
    - `find_package(LLVM)` 成功但 `eippf_protection_suite_pass` build 失败，直接 fail-closed；禁止回退 `cl.exe`。
    - 脚本必须打印最终采用的：
      - `clang-cl` 路径
      - `LLVM_DIR`
      - `runtime lib` 路径
  - workflow 不再单独设置未经验证的 `LLVM_DIR` 硬编码路径供 plugin 预构建使用。
