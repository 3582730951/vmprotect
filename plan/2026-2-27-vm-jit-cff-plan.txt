5.2 严格 Canonicalization Profile 与 Parser Parity
1. 解析前置拒绝：
- 重复 key 直接拒绝（hard-fail）。
- 非 UTF-8、含 BOM、非法 surrogate、控制字符未转义直接拒绝。
2. 数值规范化：
- 禁止 `NaN/Inf/-Inf`。
- `-0` 规范化为 `0`。
- 禁止前导零整数（`0` 除外）。
- 小数去尾零；指数记法统一小写 `e`，去 `+` 号与多余前导零。
3. 编码规范：
- UTF-8（无 BOM）+ Unicode NFC。
- 对象键按字典序排序。
- 仅允许 `\n` 作为换行。
4. Parser Parity（CI 与 Runtime 必须一致）：
- CI 签名器与 Runtime 验签器使用同一规范化版本号 `canon_profile_rev`。
- `core/tests/security/canonicalization/golden/*` 与 `adversarial/*` 必须双端产出相同 canonical bytes 与 SHA256。
- 任一差异 -> hard-fail，禁止发布。

5.3 签名机制与紧急撤销流程
1. 算法：Ed25519 detached signature；签名证据链包含 `signing_key_id + signing_key_epoch`。
2. 产物：
- `route_manifest.json`
- `route_manifest.sig`
- `route_manifest.pubid`
- `key_crl.json`（撤销列表，签名）
- `key_denylist.json`（紧急拒绝列表，签名）
3. 私钥：仅 CI KMS 可读；公钥 ID、主体标识、轮换策略、epoch 映射入库。
4. 常规轮换：90 天强制轮换，旧 key 最多并行验证 14 天，`signing_key_epoch` 每次轮换递增。
5. 紧急撤销（必须可执行，RTO <= 15 分钟）：
- T+0：安全事件立案（`security_incident_id`），冻结受影响签名 key。
- T+5m：由 `Agent 3 (Security Reviewer)` 与安全负责人双签发布 `key_denylist.json`（含 `revoked_key_ids[]`、`revoked_epochs[]`、`not_before_utc`、`reason_code`）。
- T+10m：发布 `key_crl.json` 并提升 `min_allowed_key_epoch`。
- T+15m：CI 与 Runtime 必须加载新拒绝链；命中撤销 key/epoch 立即 hard-fail，不得降级为告警。

5.4 CI 绑定校验（hard-fail）
1. 校验顺序：
- Canonicalization 结果与 hash
- manifest 签名与 `signing_key_id/signing_key_epoch`
- `key_crl.json` + `key_denylist.json` + `min_allowed_key_epoch`
- commit/tree/binary 绑定
- `critical_downgrade_approval_bindings` 的 payload hash + 双签主体/密钥隔离校验
- 可信时间锚校验（签名构建时间 + 单调时钟偏差 + 远程证明回退）
- 新鲜度、版本下限与高水位回放检查
2. 绑定规则：
- `git_commit_sha` 必须等于当前构建提交。
- `output_binary_sha256` 必须等于构建产物 hash。
- `manifest_version` 必须大于主分支最近已发布版本（防回滚发布）。
- `manifest_version >= manifest_version_floor`。
- `signing_key_id` 不得出现在 denylist；`signing_key_epoch >= min_allowed_key_epoch`。
- 双签审批必须满足：`approver_principal_ids[0] != approver_principal_ids[1]` 且 `approver_pubkey_ids[0] != approver_pubkey_ids[1]`。
- 双签公钥必须映射到不同主体（subject separation），同主体双 key 视为不合规。
- `signed_build_timestamp_utc` 签名必须可验；与 CI 当前 UTC 偏差超过 `max_clock_skew_seconds` 时必须附带有效远程时间证明，否则 hard-fail。

5.5 运行时校验（反回滚 + 新鲜度 + 撤销 + 时间锚，fail-closed）
1. 启动阶段必须执行：
- 验证 `route_manifest.sig` 与 `route_manifest.pubid`。
- 校验 `signing_key_id/signing_key_epoch`：不得命中本地拒绝链（`key_crl.json`/`key_denylist.json`），且 `signing_key_epoch >= min_allowed_key_epoch`；任一不满足立即失败。
- 校验当前二进制 hash 与 `output_binary_sha256`。
- 校验 `git_commit_sha` 与嵌入构建元数据一致。
- 校验可信时间锚：先验 `signed_build_timestamp_utc` 签名，再计算 `wall_clock_delta` 与 `monotonic_delta`；若检测到本地时钟回拨（`wall_clock_delta + max_monotonic_drift_seconds < monotonic_delta`），必须要求远程证明时间令牌，缺失或验签失败即 hard-fail。
- 校验 `generated_at_utc`：不得晚于锚定当前时间 `+max_clock_skew_seconds`，且不得早于锚定当前时间 `-valid_for_seconds`。
- 校验版本下限：`manifest_version >= manifest_version_floor` 且 `manifest_version >= kRuntimeManifestFloor`。
- 校验单调性：`manifest_version >= last_accepted_manifest_version`（持久化高水位，见 5.6）；更高版本通过后原子更新高水位。
- 校验 `critical_downgrade_approval_bindings`：payload canonical hash、双签、公钥 ID、主体隔离、过期时间全部匹配。
2. 任一失败行为：
- 进入 fail-closed：禁止启动受保护主流程并终止。
- `signing_key_id/signing_key_epoch` 撤销命中时返回 `ErrorCode::kSigningKeyRevoked`（立即硬失败，不重试主流程）。
- 时钟回拨且无有效远程证明时返回 `ErrorCode::kClockRollbackDetected`。
- 高水位篡改/回滚时返回 `ErrorCode::kManifestRollbackDetected`。

5.6 反回滚高水位持久化（防篡改）
1. 持久化对象：
- `manifest_hwm.sealed`（主槽）与 `manifest_hwm.sealed.bak`（备槽），内容字段：
- `hwm_manifest_version`、`hwm_key_epoch`、`hwm_wall_clock_utc`、`hwm_monotonic_ticks`、`monotonic_counter`、`state_mac`。
2. 防篡改封装：
- payload 采用 canonical JSON 序列化后计算 `state_mac = HMAC-SHA256(k_seal, payload)`。
- `k_seal` 来源于平台密封密钥（TPM/Keychain/DPAPI）；不可导出。
3. 单调计数策略：
- 优先硬件单调计数（TPM NV counter/可信执行环境计数器）。
- 无硬件计数器时，采用双槽日志 + 计数器递增 + MAC 校验；任一槽校验失败不自动覆盖另一槽。
4. 原子更新策略：
- 仅在 manifest 校验全通过后写入高水位；
- 流程为 `write tmp -> fsync -> rename`，并同步更新备槽；
- 新值必须满足 `new_manifest_version >= old_manifest_version` 且 `new_key_epoch >= old_key_epoch`。
5. 恢复行为（必须 fail-closed）：
- 首次启动（状态文件缺失）允许初始化主槽并记录 `monotonic_counter=1`。
- 非首次启动若主槽损坏则尝试备槽；主备均损坏或计数器回退时，返回 `ErrorCode::kHighWatermarkStateCorrupt` 并终止。
- 禁止“清空状态后继续运行”；仅允许加载双签恢复包 `hwm_recovery_bundle.json`（Reviewer+Owner 双签，且主体/密钥仍需隔离）。

六、遥测/分类输入严格 fail-closed（与 2.2/2.5 对齐）
6.1 输入状态分类
1. `OK`：文件存在、schema 完整、签名/hash 全通过。
2. `ABSENT_HISTORY`：历史遥测为空，但 `X_static` 完整且元数据合法。
3. `CORRUPT`：签名错误、hash 不匹配、字段缺失、非法数值、时间戳非法。

6.2 处理矩阵（强制）
1. `OK`：使用 `Profile-H`（`X_static ∪ X_dyn`）进行模型推断与路由。
2. `ABSENT_HISTORY`：仅允许进入 `Profile-N` 与 `2.5` 冷启动路由；`x8/x11/x12` 缺失视为正常，不得报错。
3. `CORRUPT`：
- CI：立即 hard-fail，禁止生成新路由；
- Runtime：拒绝加载路由并 fail-closed。

6.3 禁止行为
1. 禁止 silent fallback 到 `NONE`。
2. 禁止在输入损坏时继续沿用“未签名最新路由”。
3. 禁止因缺少 `X_dyn` 就阻断 `ABSENT_HISTORY` 路径。

七、Critical 路径防弱化与加密审批绑定
7.1 强制路由底线
1. `critical=1` 的函数禁止 `strategy -> NONE`。
2. `Hot + Critical` 仅允许 `JIT_ENCLAVE + CFF_FULL + BCF_STRONG`。
3. `Non-Hot + Critical` 最低保护线为 `VM_STRONG + CFF_LITE + BCF_LITE`。

7.2 审批工作流（双签 + 主体/密钥隔离 + 有效期）
1. 申请载荷文件：`core/runtime/policy/critical_downgrade_requests/<approval_id>.json`（Canonical JSON）。
2. 必填字段：
- `approval_id`
- `function_id`
- `from_strategy` / `to_strategy`
- `commit_range`
- `justification`
- `perf_evidence_report_sha256`
- `risk_assessment`
- `issued_at_utc` / `not_after_utc`
- `ticket_id`
- `security_reviewer_principal_id`
- `security_owner_principal_id`
- `security_reviewer_pubkey_id`
- `security_owner_pubkey_id`
3. 双签文件（detached）：
- `<approval_id>.security_reviewer.sig`
- `<approval_id>.security_owner.sig`
4. 隔离硬规则（CI + Runtime 同步执行）：
- `security_reviewer_principal_id != security_owner_principal_id`
- `security_reviewer_pubkey_id != security_owner_pubkey_id`
- 两把 key 必须映射到不同主体证书链（subject DN 或等价主体 ID 不得相同）。

7.3 审批与 manifest 的加密绑定（修复“仅 ID”）
1. CI 在路由生成时必须计算并写入 manifest：
- `approval_payload_sha256 = SHA256(canonical(approval_payload))`
- `approval_sig_sha256[] = SHA256(signature_files)`
- `approver_principal_ids[] = [security_reviewer_principal_id, security_owner_principal_id]`
- `approver_pubkey_ids[] = [security_reviewer_pubkey_id, security_owner_pubkey_id]`
2. Runtime 验证顺序：
- 读取 manifest 的 binding；
- 加载对应 approval payload 与签名文件；
- 复算 payload/sig hash 与 manifest 对比；
- 验证双签公钥 ID 与签名有效性；
- 验证主体隔离与密钥隔离（principal 不同 + key 不同 + 证书主体不同）；
- 校验 `not_after_utc` 与 `commit_range`。
3. 任一不匹配：fail-closed。
4. 错误码细分：
- 哈希/签名不匹配：`ErrorCode::kCriticalApprovalBindingFailed`
- 主体或密钥未隔离：`ErrorCode::kApprovalSeparationFailed`

7.4 审批生效边界
1. 批准有效期最长 30 天，过期自动失效。
2. 仅对指定 `function_id + commit_range` 生效，不可泛化。

八、CI 安全 hard-fail 门禁（含 BCF、反回滚、撤销链与时间锚）
8.1 CI 阶段顺序（固定）
1. Stage S1：输入完整性校验（telemetry/model/dataset/schema）。
2. Stage S2：热点分类复现实验与 F1 验证。
3. Stage S3：路由生成与策略合法性校验（含 Hot 必须 CFF+BCF）。
4. Stage S4：Canonicalization/Parser Parity + manifest 签名校验。
5. Stage S5：密钥信任链校验（`key_crl.json`、`key_denylist.json`、`min_allowed_key_epoch`）。
6. Stage S6：commit/tree/binary/approval hash 绑定 + 双签主体/密钥隔离校验。
7. Stage S7：可信时间锚校验（签名构建时间 + monotonic delta + 远程证明回退）。
8. Stage S8：高水位持久化防篡改演练（主备槽损坏、计数器回退、恢复包校验）。
9. Stage S9：fail-closed 单元/集成测试。
10. Stage P1：性能预算 gate（第三章）。

8.2 Hard-fail 条件（任一触发即阻断）
1. 签名校验失败或公钥 ID 不在信任列表。
2. `signing_key_id` 命中 denylist/CRL，或 `signing_key_epoch < min_allowed_key_epoch`。
3. `git_commit_sha` / `binary_sha256` 任一不匹配。
4. 热点 F1 未达门槛。
5. 任意 Hot 函数未启用 `CFF+BCF`。
6. `C_bcf_hat` 字段缺失或非法。
7. telemetry/classification 输入为 `CORRUPT`。
8. Critical 降级缺失 payload hash 绑定、双签不通过，或主体/密钥未隔离。
9. `manifest_version` 回退、低于 floor，或 `generated_at_utc` 超窗口。
10. 高水位状态主备槽均损坏、MAC 失败或 monotonic counter 回退。
11. 检测到本地时钟回拨且无有效远程证明时间令牌。
12. 紧急撤销发布后 15 分钟内 CI/Runtime 未拉取到新拒绝链。
13. fail-closed 测试失败。
14. 性能预算不达标（`Budget_pred` 或 `Budget_real` 超标，或 Critical `p95 > 0.15`）。

8.3 禁止绕过条款（硬要求）
1. 所有 hard-fail gate 一律禁止 bypass，不存在白名单开关。
2. 禁止 `--allow-insecure`、`--skip-security-gates`、`FORCE_MERGE` 类覆盖参数。
3. gate 执行器不可用或超时视同失败，不得降级为告警。

8.4 必须归档产物
1. `build/perf/perf_gate_result.json`
2. `build/perf/security_gate_result.json`
3. `build/perf/hotspot_report.json`
4. `build/perf/route_policy_report.json`
5. `build/perf/route_manifest.json` + `.sig`
6. `build/perf/critical_downgrade_audit.json`
7. `build/perf/key_revocation_audit.json`
8. `build/perf/time_anchor_validation_report.json`
9. `build/perf/hwm_persistence_audit.json`

九、分阶段实施（Phase 28-34）
Phase 28：C 成本模型 v2 与 `C_bcf` 落地
- 交付：特征提取器、`Profile-H/N` 推断器、`c_cost_predictions.json`。
- 验收：`C_bcf_hat` 全量产出；`ABSENT_HISTORY` 可无 `x8/x11/x12` 运行。
- 安全：模型文件签名校验接入。

Phase 29：预算公式与 BCF gate 固化
- 交付：`Budget_pred/Budget_real` 脚本、`coverage_hot_cff_bcf` 统计。
- 验收：重复测量波动 `<=3%`，且 Hot 覆盖率达 100%。
- 安全：环境锁定参数进入审计日志。

Phase 30：热点验证可复现化
- 交付：冻结数据集 r2、标注规则、F1 评估脚本。
- 验收：Macro-F1/critical F1 达标。
- 安全：样本脱敏检查通过。

Phase 31：Manifest 强规范化 + 反回滚/新鲜度 + 高水位防篡改
- 交付：Canonicalization 实现、parser parity 测试、`manifest_hwm.sealed` 主备槽、MAC 校验与 monotonic counter 逻辑。
- 验收：篡改/回滚/过期样本 100% 被阻断；主槽损坏时可受控回退备槽；主备同时异常时 fail-closed。
- 安全：禁止“清空状态继续运行”，仅允许双签恢复包流程。

Phase 32：Critical 降级审批加密绑定 + 双签主体隔离
- 交付：approval payload schema、双签工具、主体/密钥映射表、manifest binding 与 runtime 验证。
- 验收：仅 ID 无 hash 绑定、同主体双签、同 key 双签均无法通过 CI/Runtime。
- 安全：审批过期、签名不匹配、主体隔离失败立即阻断。

Phase 33：密钥生命周期与紧急撤销并线
- 交付：`key_crl.json`、`key_denylist.json`、`min_allowed_key_epoch` 门禁、撤销事件自动化脚本与审计日志。
- 验收：撤销 key/epoch 在发布后 15 分钟内被 CI 与 Runtime 同步拦截（立即 hard-fail）。
- 安全：撤销链拉取失败按失败处理，不得降级运行。

Phase 34：CI 安全硬门禁并线（不可绕过）
- 交付：S1-S9 + P1 全门禁流水线与归档产物。
- 验收：安全 gate 与性能 gate 全绿方可合并。
- 安全：不提供任何 hard-fail bypass 机制。

十、最终验收总线（必须同时满足）
1. 冷启动：`ABSENT_HISTORY` 路由不依赖 `x8/x11/x12`，且与 `2.2/2.5/6.2` 一致。
2. BCF：`C_bcf_hat`、Hot 路径 `CFF+BCF`、BCF gate 均启用且生效。
3. 门禁：安全 hard-fail gate 全部不可绕过。
4. 新鲜度：runtime 完成 `manifest_version` 单调校验、`generated_at_utc` 窗口校验、`version floor` 校验并 fail-closed。
5. 规范化：重复 key 拒绝、数值规范化、UTF-8/NFC 编码配置、parser parity 全通过。
6. 审批：Critical 降级绑定 payload hash + 双签并由 runtime 验证，禁止仅 ID 通过。
7. 撤销：存在紧急撤销 SOP；Runtime 启动时强制执行 `CRL/denylist/key_epoch` 校验，命中即 `kSigningKeyRevoked` 立即 hard-fail。
8. 高水位：`manifest_hwm.sealed` 采用密封存储 + MAC + 单调计数策略；篡改、回滚、状态损坏恢复失败必须 fail-closed。
9. 双签隔离：双签审批必须来自不同 principal 且不同 key，并在 CI 与 Runtime 双端强制检查。
10. 时间锚：采用“签名构建时间 + monotonic delta + 远程证明回退”；本地时钟回拨无法绕过。
