## Task Summary
- 本轮只修 `sample-suite-eval` 在 run `23967734619` 新暴露的 3 个真实阻塞点，不扩范围：
  1. Windows workflow 的 LLVM preflight 发现逻辑错误，导致 bundled/fallback 都不能稳定落到可用根。
  2. Linux/Android 平台脚本在 `-x c` 后没有对 helper `.o` 做语言复位，导致对象文件被当前语言状态当成源码。
  3. macOS/iOS 平台脚本用 `clang++` 驱动 `.c` 样例但没有显式语言边界，触发 `treating 'c' input as 'c++'` 并走到 Apple clang frontend crash。
- 本轮目标是把这 3 个失败点关掉，并把 validator 升级到能稳定阻止同类回归。

## Architecture Contract
### Objective
- 把 workflow 的职责重新收紧为环境发现、脚本调度、机器校验。
- 把语言边界修复固定在平台脚本内，不让 workflow 或 wrapper 成为第二事实来源。

### System Boundaries
- Windows
  - workflow 负责找到唯一可用 LLVM 根并导出环境。
  - `build_windows.ps1` 继续消费环境并产出既有报告，不改其内部构建语义。
- Linux/Android
  - `build_linux_android.sh` 负责所有 link args 真相。
  - workflow validator 只读取 sidecar/report 做 token 状态机校验。
- macOS/iOS
  - `build_macos_ios.sh` 负责最终 wrapper link 真相。
  - workflow validator 只检查 `-x c -> source -> -x none -> helper.o` 合约与现有 provenance。

### Invariants
- report key 集合不变。
- artifact upload/download 布局不变。
- helper 编译 flags 不变。
- Windows job 继续使用 `clang-cl`。
- macOS job 继续使用 `xcrun --sdk iphonesimulator --find clang++` 作为最终 wrapper `--compiler`。

### Failure Handling
- Windows preflight 必须在找不到任何满足条件的 LLVM 根时 fail-closed，并列出所有尝试的 candidate root。
- Linux/Android 与 macOS/iOS validator 只要发现 helper 对象仍处于显式源语言状态，直接 fail。
- 所有新增 gate 都必须是机器可判定，不能依赖日志人工阅读。

## Regression Perimeter
### Root Cause
- Windows
  - workflow 安装后直接假定 `C:\Program Files\LLVM\lib\cmake\llvm`，没有重跑发现逻辑。
- Linux/Android
  - `-x c` 没有在 helper `.o` 前显式 reset，`.o` 被前端按源码读入。
- macOS/iOS
  - `clang++` 处理 `.c` 源时没有 `-x c` 和 `-x none` 语言边界。

### Directly Changed Paths
- `/workspace/.github/workflows/sample-suite-eval.yml`
- `/workspace/core/tools/sample_suite/build_linux_android.sh`
- `/workspace/core/tools/sample_suite/build_macos_ios.sh`

### Adjacent Paths
- `/workspace/core/tools/sample_suite/build_windows.ps1`
- `/workspace/core/wrapper/eippf_cc.py`

### Unchanged Guarantees
- helper 构建语义不改。
- toolchain report key schema 不改。
- `protect-and-evaluate` job 不改。

### Second-Order Risks
- 三条工程轨都碰同一个 workflow 文件，容易冲突。
- validator 如果只用字符串匹配而不做 token 状态机，会漏掉顺序错位。
- Windows runner 可能存在第三种 LLVM 布局，preflight 必须报告完整 candidate 列表。
- 若状态机只枚举常见源语言而不把“任何显式 `-x <lang>` 命中对象输入”一律判死，会留下侥幸过关路径。

## Decision Log
- D1
  - Windows 本轮优先修 workflow preflight，不碰 `build_windows.ps1`。
  - 理由：当前失败发生在脚本启动前，且现有 Windows 报告 schema 已足够支撑 validator。
- D2
  - Linux/Android 与 macOS/iOS 都采用显式 `-x c` / `-x none` 语言边界，不改 wrapper。
  - 理由：失败根因在平台脚本参数序列，不在 wrapper 语义。
- D3
  - validator 必须升级为 token 状态机，不接受简单字符串包含。
  - 理由：只有状态机才能稳定拦截顺序错误与带空格路径。

## Open Risks
- Windows runner 可能返回多个候选 LLVM 根；preflight 必须按顺序选第一个同时满足 `clang-cl.exe + LLVMConfig.cmake` 的根。
- macOS 若 Apple clang 本身存在 frontend bug，本轮也只能通过显式语言边界规避，不会改变 Apple toolchain 行为。
