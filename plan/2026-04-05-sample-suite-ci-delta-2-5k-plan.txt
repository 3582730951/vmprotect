Root Cause Candidates:
- P0:
  - 当前 Windows helper 允许 `TARGET LLVM` 或较宽的 `llvm_map_components_to_libnames(...)` 结果直接进入 pass plugin 链接面，把 pinned LLVM18 在 Windows 上的 DIA SDK 依赖一并带入。
  - 证据：
    - `PassPluginCompat.cmake` 当前仍有 `if(TARGET LLVM) target_link_libraries(${name} PRIVATE LLVM)`。
    - 当前组件列表包含 `ipo`、`Instrumentation`、`IRReader`，但 pass 源文件并未包含对应头族。
  - 如何证伪：
    - 检查生成的 Windows link line 或 Ninja 依赖；若去掉 monolithic/宽组件后 `diaguids.lib` 消失且 DLL 仍能成功链接，则 P0 成立。
- P1:
  - 触发 DIA 依赖的不是 monolithic `LLVM` 本身，而是当前宽组件集合中的一个或多个组件把 `diaguids.lib` 传递进来。
  - 优先怀疑：
    - `ipo`
    - `Instrumentation`
    - `IRReader`
  - 如何证伪：
    - 把 Windows pass plugin 改成只解析固定最小组件 allowlist；若 `diaguids.lib` 消失，则 P1 成立。
- P2:
  - pinned LLVM18 的 Windows CMake package 即使走组件解析，也可能仍把 `diaguids.lib` 暴露为某个 resolved item。
  - 如何证伪：
    - 在 helper 中对 resolved Windows link items 做 denylist 检查；若 denylist 被触发，则 P2 成立并 fail-closed。
- P3:
  - runner 缺少某个 VS DIA SDK 路径只是表象，真实问题仍是 pass plugin 不该依赖 DIA。
  - 如何证伪：
    - 若缩小 LLVM 组件后完全不再请求 `diaguids.lib`，则环境不是主因。
  - 处理优先级：
    - 最低。
    - 本 delta 明确禁止通过“补环境”修复它。
