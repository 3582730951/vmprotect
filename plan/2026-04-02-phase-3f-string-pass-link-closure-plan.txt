Architecture Contract:
- Root Contract:
  - `Phase 3d` 已把 `string_protection_pass_test` 注册回 build/test 图。
  - `Phase 3e` 已把测试源码的 GEP 二义性 compile blocker 收口。
  - 当前唯一 blocker 是 `/workspace/core/tests/passes/CMakeLists.txt` 的 fallback 链接策略把测试目标接成了组件名 `-lLLVMCore`、`-lLLVMSupport` 等，但当前环境下这些组件库名无法直接被链接器解析。

    - `llvm-config --libdir` 输出 `/usr/lib/llvm-14/lib`
    - `ip_weaver_ir` 当前最终 link line 使用 `/usr/lib/llvm-14/lib/libLLVM-14.so.1`
- 固定修法:
  - 只允许修改 `/workspace/core/tests/passes/CMakeLists.txt`。
  - 必须保持以下现状不变：
    - `add_executable(string_protection_pass_test ...)`
    - `add_test(NAME string_protection_pass_test ...)`
    - `string_protection_pass_test.cpp` 与 `passes/src/StringProtectionPass.cpp` 源列表
    - `EIPPF_LLVM_INCLUDE_DIRS` include 接线
  - `if(TARGET LLVM)` 分支保留不动。
  - `elseif(COMMAND llvm_map_components_to_libnames)` 分支不允许继续使用 `llvm_map_components_to_libnames(...)` 的结果去链接组件名。
  - 该 fallback 分支必须改成如下固定结构，不允许 Engineer 自己选择其他方案：
    1. 保留 `elseif(COMMAND llvm_map_components_to_libnames)` 这个分支壳，不新增新的顶层条件分支。
    2. 在该分支内新增：
       - `find_program(EIPPF_LLVM_CONFIG_EXECUTABLE NAMES llvm-config REQUIRED)`
    3. 在该分支内新增：
       - `execute_process(COMMAND "${EIPPF_LLVM_CONFIG_EXECUTABLE}" --libdir OUTPUT_VARIABLE EIPPF_STRING_PROTECTION_TEST_LLVM_LIBDIR OUTPUT_STRIP_TRAILING_WHITESPACE)`
    4. 在该分支内新增：
       - `find_library(EIPPF_STRING_PROTECTION_TEST_LLVM_MONOLITHIC NAMES LLVM-14 LLVM HINTS "${EIPPF_STRING_PROTECTION_TEST_LLVM_LIBDIR}" NO_DEFAULT_PATH REQUIRED)`
    5. 在该分支内把原来的 component `target_link_libraries(...)` 替换为：
       - `target_link_libraries(string_protection_pass_test PRIVATE "${EIPPF_STRING_PROTECTION_TEST_LLVM_MONOLITHIC}")`
    6. 不允许在该计划范围内使用：
       - `llvm-config --libs` 的输出直接做字符串拆分链接
       - `target_link_directories(...)`
       - 修改 `ip_weaver_ir`
       - 修改任何生产模块
  - 这次修法的目标是让 `string_protection_pass_test` 的 fallback link line 改成 monolithic LLVM 库的绝对路径形式，对齐当前 `ip_weaver_ir` 的已知可用样式。

  - Q4 / 具体链接修法是什么
