**Task Card 2: Unsafe-Shell Redlight 覆盖补齐**
- Owner: `Agent 2 (Engineer)`
- Files: `/workspace/core/tests/script_guard/script_launcher_security_edge_test.cpp`
- Change: 在现有 `set -x` 与 `source` 基础上，补齐计划 G2 redlight 缺失的四类脚本 fixture，并明确断言 bundler 在生成 bundle/manifest 前即拒绝。
- Required Cases: `set -o xtrace`、行首 `. ./other.sh`、`$0`、`${BASH_SOURCE}` 或 `${BASH_SOURCE[0]}`。
- Guardrail: 保留现有 safe script 成功路径与 env scrub 断言，避免 redlight 扩散成误拦截。
- Done When: G2 列出的 redlight 全部有独立 fixture 和独立失败断言。

**Task Card 3: Executable Provider `stderr` 泄漏封堵**
- Owner: `Agent 2 (Engineer)`
- Files: `/workspace/core/script_guard/src/external_key_provider.cpp`、`/workspace/core/tests/script_guard/script_launcher_success_test.cpp`、`/workspace/core/tests/script_guard/script_guard_smoke_test.cpp`
- Change: 对 `read_provider_response_from_executable()` 增加独立 `stderr` 处理，不再让 provider child 继承父进程 `stderr`。最小策略为“捕获并阻断，不转发”；若 provider `stderr` 非空，则判定该 provider 违规并 `fail-closed`。
- Security Rule: provider 协议只允许通过 `stdout` 返回；`stderr` 中任意字节都不能出现在父进程 `stdout`、`stderr`、bundle、manifest、测试输出文件或临时副产物中。
- Success Probe: 增加一个安静的 `executable_adapter` happy path，证明封堵后合法 adapter 仍可工作。
- Failure Probe: 增加一个会在 `stderr` 打出 secret marker 但 `stdout` 返回合法协议的 adapter fixture，断言 bundler/launcher 失败，且 marker 不出现在任何父可见输出中。
- Done When: `executable_adapter` 既有合法成功路径仍可通，带 `stderr` 的 adapter 被 fail-closed 且无泄漏。

**Task Card 4: Shell Strict Audit 真实断言闭口**
- Owner: `Agent 2 (Engineer)`
- Files: `/workspace/core/tests/tools/artifact_audit_strict_test.cpp`
- Change: 对已写出的 `shell_bundle_artifact` 和四类 shell manifest 执行真实 `run_audit(... --manifest ... --strict)`，不再停留在 fixture 写入阶段。
- Required Assertions: success manifest 返回 `0` 且报告包含 `\"strict_failures\": []`；bad-gate manifest 返回非零且报告包含 `shell_gate_failed`；unsafe manifest 返回非零且报告包含 `shell_unsafe_feature_present`；leak manifest 返回非零且报告包含 `shell_plaintext_leak_indicator`。
- Attribution Rule: shell success fixture 必须保持 denylist-clean，避免 strict failure 被无关规则抢占；若 leak fixture 同时命中 `shell_gate_failed`，可作为附加断言，但不能替代 `shell_plaintext_leak_indicator` 的专门断言。
- Done When: shell strict gate 的 success/failure token 都来自真实 audit 执行，而不是字符串静态拼装。
